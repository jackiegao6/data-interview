Kafka 是如何 高效读取数据的

1.顺序io：Kafka的消息存储采用日志文件形式，消息不断追加到日志文件末尾
2.零拷贝，靠sendfile系统调用 磁盘->内核缓存->网卡 ，只需要复制两次，而且CPU不参与拷贝
                            Page Cache->网卡 优先页缓存中查数据，若命中则直接读取，减少了磁盘IO操作
kafka不直接操作磁盘，而是操作Page Cache，写入时写到Page Cache就算成功，由OS异步刷新到磁盘；读取时如果数据还在Page Cache里，就直接从内存返回；
3.
5.生产者是批量发送的、消费者是批量读取的，而不是一条一条拉，减少了RTT


零拷贝是如何做到的
传统方式
1.磁盘->page cache(CPU 不参与 DMA完成)
2.page cache -> kafka 进程内存（CPU参与）
3.kafka内存 -> socket 内核缓冲区（CPU参与）
4.socket 内核缓冲区 -> 网卡 (CPU 不参与 DMA完成)
数据在内存搬来搬去，无效劳动，kafka根本不需要修改数据内容


Kafka写入时怎么保证不丢数据
1.生产者发送完消息时设置ack参数，规定broker写入成功的行为
    ack=0 发出去就算成功，不等任何回应
    ack=1 leader分区收到就算成功
    ack=-1 所有副本分区收到才算成功
2.重试机制 规定时间内没收到ack就重试
3.幂等性 防止重试导致消息重复写入 
    开启幂等后 acks硬性覆盖为 -1 原因是保证故障时的幂等语义 确保生产者收到acks 时 所有副本都已同步offset
4. broker端的副本同步机制
5. broker端的刷盘策略
    不建议依赖它们做可靠性保证，更依赖副本机制
6. 消费者通过维护偏移量来记录已消费消息的位置


Kafka的文件存储机制是怎样的
1.目录：每个Topic-分区是一个文件夹
2.分段：文件夹里切成多个Segment文件
3.文件对：每个Segment由.log（存消息体，顺序写）和.index（稀疏索引，加速查）组成
4.清理：过期（默认7天）删除

Pull模式：Consumer主动从Broker拉取消息
1.Consumer 可以自主决定拉取消息的速率，根据自身的消费能力来拉取数据，避免因处理不过来而崩溃
2.Consumer 可以自主决定拉取的批量大小
3.会有空轮询问题：通过epoll机制解决

Kafka的分区分配策略有哪些
生产者端：消息写入哪个分区
1.指定分区
2.按消息的key哈希（默认）
3.轮询
4.无key时粘性，填满一个分区后再换下一个（默认）
消费者端：分区如何分配给消费者组里的消费者
1.轮询：负载均衡，再平衡时迁移开销大
2.粘性分配：在保证负载均衡的前提下，尽可能保持消费者现有的分区分配不变
